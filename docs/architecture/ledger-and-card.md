# Ratchet — Ledger, Improvement Curve & Level-Up Card

**Status:** Draft v0.1 · Read `skill-schema.md` (§5 LedgerEntry) and `proof-gate.md` (§4 yardstick) first.

> The ledger is the **proof made visible**, and the level-up card is the **viral artifact** — the
> 20-second "watch it level up" payoff (Inventory B4, G3). It is treated as a **feature**, not an
> afterthought: it is how a skeptical second persona trusts the product and how the first persona
> shares it. The card never shows anything the ledger can't back with a ProofRun manifest (AGENTS.md
> invariant 7).

---

## 1. What the ledger is

An **append-only** record of every promotion/retirement/quarantine/revalidation event, each a
`LedgerEntry` (`skill-schema.md` §5). Append-only because the "forward-only ratchet" needs an
immutable history — you can read how the agent got better, and no past entry is rewritten.

```
.ratchet/ledger/
├─ 0001-2026-06-18-promoted-01J....json   # one file per event, zero-padded sequence + date
├─ 0002-2026-06-20-promoted-01J....json
├─ 0003-2026-06-22-retired-01J....json    # staleness retirement (proof-gate §8)
└─ 0004-2026-06-22-promoted-01J....json
```

Each `LedgerEntry` (per schema) carries: `event`, `skillId`, `proofRunId`, `at`, `cumulativeSkills`,
and `benchmarkScore` (the running score on the stable yardstick). The SQLite index
(`.ratchet/index.sqlite`) mirrors these for fast queries; the JSON files remain the source of truth
for portability (`overview.md` §5).

```
event → LedgerEntry (append) → curve point (cumulativeSkills, benchmarkScore, at) → card render
```

## 2. The improvement curve

The curve is the time series of `benchmarkScore` measured on a **stable yardstick task set** — the
same held-out benchmark every time, so the curve is *comparable across the agent's whole life*
(`proof-gate.md` §4). Without a fixed yardstick, each point would use a different ruler and the curve
would be meaningless.

- **Yardstick definition:** a frozen, versioned task set (`proof.yardstickId` in `config-schema.md`),
  separate from any individual skill's proof tasks. Re-scored after each promotion. Its id and
  version travel in every curve point so two curves are only ever compared on the *same* yardstick +
  version.
- **What's plotted:** x = event sequence or `at` timestamp; y = `benchmarkScore` on the yardstick.
  Promotions step it up; a retirement may step it down (and that's honest — the curve shows real
  state, including decay).
- **Leakage guard:** the yardstick is never used as a candidate's own proof task (would be
  teaching-to-the-test, `proof-gate.md` §5), keeping the curve an independent measure.

```
benchmarkScore
  ^
  │                                   ┌─────●  0.91
  │                          ┌────────┘
  │                 ┌────────┘ ●  (promotion adds a verified skill)
  │        ┌────────┘
  │   ●────┘
  └────────────────────────────────────────────▶ events / time
      ↑ cold start (0 skills)        ↑ each step = a PASSED proof gate
```

## 3. The level-up card (the viral artifact)

A single shareable image auto-generated from the ledger + curve. Generated by `ratchet ledger --card`
(see `cli-mcp-interface.md` §5) or the `ratchet.card` MCP tool.

### 3.1 Visual concept

```
┌──────────────────────────────────────────────────────────┐
│  🔧  RATCHET                          claude-code · @user  │
│                                                            │
│   LEVEL 7   ·   12 skills earned   ·   0 regressions       │
│                                                            │
│   benchmark on "default-yardstick v3"                      │
│        ┌───────────────────────────────────────●  +27%    │
│        │                         ┌───────●──────┘          │
│        │            ┌────●───────┘                         │
│        ●────────────┘                                      │
│        cold start ──────────────────────▶ today           │
│                                                            │
│   Latest skill earned:                                     │
│   ✅ "Run db migrations before tests"   +18%, 7 trials     │
│      CI excludes 0 · 0 regressions · receipt 01J9Z5N4…     │
│                                                            │
│   every skill earned · proven vs baseline · never backslid │
│   ratchet.dev · verify: npx ratchet ledger show 01J9Z5M2…  │
└──────────────────────────────────────────────────────────┘
```

The card's spine is the curve; the hook is the headline (Level, skills earned, **0 regressions**);
the credibility is the receipt id anyone can re-verify. "XP for your AI" is legible at a glance to the
vibe coder, while the receipt + yardstick satisfy the researcher (PRD §3 dual personas).

### 3.2 Fields shown on the card

| Field | Source | Why it's there |
|---|---|---|
| Level | derived: `f(cumulativeSkills, benchmarkScore)` | the "XP" hook; gamified summary |
| Skills earned | `cumulativeSkills` (latest entry) | proof of real, verified learning |
| Regressions | count of regression events (target: 0) | the "never backslides" promise made visible |
| Improvement % | `(latest − first) / first` benchmarkScore on the yardstick | the headline number |
| Yardstick id + version | `proof.yardstickId` | so the curve is comparable / not cherry-picked |
| Curve sparkline | `benchmarkScore` over events | the "watch it level up" visual |
| Latest skill | most recent `promoted` entry → `Skill.name` | concreteness |
| Latest skill measurement | deciding `ProofRun.measurement` (delta, trials, significance) | receipts, not vibes |
| Receipt id | `proofRunId` | anyone can re-verify via `ratchet ledger show` |
| Host + handle | `config.host.adapter`, `createdBy` (handle only) | attribution, **no PII** |

**Never on the card:** raw transcripts, secrets/PII, un-proven ("candidate") skills, or any number
not backed by a manifest (AGENTS.md invariants 5, 7).

### 3.3 The "Level" function

Level is a presentation-layer convenience, not a trust signal — trust lives in the ProofRuns. A
simple, stable mapping (so two users' levels are comparable):

```
level = floor( sqrt(cumulativeSkills) + improvementBonus )
improvementBonus = clamp( (latestBenchmark − firstBenchmark) / firstBenchmark , 0, 3 )
```

It is deliberately monotonic with verified skills + measured improvement, and never rewards
un-promoted skills.

## 4. Output formats

`ratchet ledger --card --format <svg|md>`:

- **`svg`** (default) — themeable vector; the canonical artifact (crisp on every platform).
- **`md`** — a markdown block (table + ASCII sparkline) for READMEs and the vault `index.md`.

All three render from the **same** ledger query, so they always agree. The SVG uses CSS variables for
theming and embeds the yardstick id + receipt id as visible text (not just metadata) so the card is
self-verifying.

## 5. Generation pipeline

```
ledger entries ──┐
                 ├─▶ resolve curve (yardstick-scored points) ──▶ render(card) ──▶ svg|png|md
yardstick meta ──┘            │                                      │
deciding ProofRun ────────────┘ (latest skill + measurement + id) ───┘
```

1. Read the append-only ledger (+ SQLite index for speed).
2. Build the curve from `benchmarkScore` points on the configured yardstick (+version).
3. Pull the latest `promoted` entry's deciding `ProofRun.measurement` and `proofRunId`.
4. Render. Every visible claim traces to a ledger entry + manifest; nothing is fabricated.

## 6. Privacy & integrity

- The card and ledger contain **no** secrets/PII (redacted upstream at distillation; AGENTS.md
  invariant 5). The only identity shown is a handle.
- Curves are only comparable on the same yardstick id + version — the card states both, so a shared
  card can't silently mislead.
- Because the ledger is append-only and each entry references a ProofRun manifest, a card is a
  **verifiable** claim: a skeptic can re-run `ratchet ledger show <skillId>` and re-verify the receipt.
